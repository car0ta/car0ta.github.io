prvd 项目流程分析
2020-01-28

简介

prvd 是一个 PHP 运行时漏洞检测系统，该项目依赖于 xmark 扩展，并由 sentry 提供客户端界面。prvd 有两种模式——taint 和 payload，payload 模式一直开启，taint模式选择开启。taint 模式在 source 点上打上标记，在 filter 中传递这个标记，在 sink 点检测标记。payload 模式在原始数据末尾加上 payload，不用关心数据传递过程而直接在 sink 点观测。

安装

  1. git clone 到一个非 web 目录下

  2. 在 prvd 目录下安装依赖

  3. 下载编译 xmark

    xmark 是 PHP7 的一个扩展，基于 PHP 的基本功能有：

    • 对字符串变量打标记
    • hook 绝大多数函数/类
    • hook 部分 opcode

    安装步骤如下：

  4. 更改 php.ini 的配置文件

    • auto_prepend_file 配置成 prvd/src/Entry.php
    • extension 配置成 xmark.so 路径
    • 其余的配置直接 copy 项目中 prvd.ini 的内容

  5. 前往 https://sentry.io 注册一个账号，并安装 sentry 库

    新版：composer require sentry/sdk:2.1.0

    旧版：composer require "sentry/sentry" （prvd 项目使用旧版）

  6. 编辑配置文件 prvd/src/Config.php

项目结构

prvd 目录结构

src 目录结构

Entry.php

先检测xmark扩展是否加载

用重写函数（前缀prvd_）替换危险函数（filter或sink），非危险函数则不重写

给 GET, POST, COOKIE, FILES, REQUEST 变量都打上标记，给SERVER变量开头是 HTTP_ 的打上标记

加载 sink、filter、opcode 及其他相关文件

Utils.php

  • prvd_xcheck —— 检查变量是否打上标记

  • prvd_xmark —— 给变量打上标记

  • prvd_check_callback —— 检查回调函数是否被打上标记，是则捕获漏洞

    定义数组 $calls，键名为危险函数（或类中的方法），键值为回调函数在参数中的相对位置 。检查回调函数名是否被打上标记 xtanzi，如果是数组（类中的方法）则检查前两个元素（类名和方法名）是否被打上标记。

  • prvd_check_dynamic_call —— 检查动态回调函数名是否可控

    动态回调函数即可变函数 —— 将函数名赋值给变量。检查函数名是否被打上标记 xtanzi，如果是数组（类中的方法）则检查前两个元素（类名和方法名）是否被打上标记。

  • prvd_check_path —— 检查 path 相关变量（路径、文件名等）

    如果 path 中包含 ../xtanzi 或者 ..\xtanzi ，则捕获漏洞。例如在dirname、basename 等函数中，如果参数 path 可控，则会导致目录穿越。

  • prvd_check_sqli —— 检测是否存在 SQL 注入

    通过调用 prvd_detect_sql_injection 函数来检测 SQL 语句是否异常。

    检测逻辑：对整个 SQL 句柄逐个字符进行分析，跳过空白字符，跳过被转义的反引号、单引号和双引号，如果出现未被转义的单引号、双引号、反引号则检测是否有相同字符闭合，若闭合则继续判断，若一直到句柄末尾都没有闭合则返回 true，进而上报漏洞。如果出现 /* 则检测是否有 */ 与之闭合，若注释符匹配则继续判断，若不匹配则一直分析到句柄末尾，检测注释中是否含有 xtanzi 标记，有则上报漏洞。如果出现 -- 的注释符，则跳过整个句柄。对于句柄中的关键字，检测是否被打上标记，若含有 xtanzi 标记则上报漏洞。

    缺陷：对于 -- 的注释符默认整个 SQL 句柄安全，实则存在处理缺陷。例如句柄 select * from user where id = 1--2 union select * from user;

    改进：对 -- 注释符后的内容检测是否含有 xtanzi 标记，如果存在则认为存在 SQL 注入。

  • prvd_check_rce —— 检测是否存在命令执行

    通过调用 prvd_detect_cmd_injection 函数来检测 cmd 语句是否异常。

    该项目只处理了逃脱引号，此处可以扩充，对更多可能的情况进行处理。

  • prvd_check_xss —— 检测是否存在 xss

    只是简单的检测字符串是否被打上标记来判断是否存在 xss，有待扩充。

  • prvd_check_serialize —— 检测是否存在反序列化（未实现）

    通过调用 prvd_detect_unserialize 函数来检测序列化后的字符串是否异常。

  • prvd_check_ssrf —— 检测是否存在 ssrf

    先解析url，如果没有主机名则说明是本地请求，检测url是否被打上标记；存在主机名就检测url中主机名是否被打上标记。

filter

filter 目录下均为函数过滤处理的文件，以 dirname 函数为例我们分析一下处理逻辑。

使用 call_user_func 回调函数得到 dirname 函数处理结果，检测 dirname 函数的 path 参数是否被打上标记，如果是则结果也打上标记。

filter 目录下的其他函数的处理逻辑与其相同，一句话总结就是： 如果函数中的可控参数带有标记，那么函数处理结果也打上标记。

opcode

我们知道，在 PHP 中可以从两个层面 hook 函数，一是通过修改 zend_internal_function 的 handler 来 hook PHP 中的内部函数，handler 指向的函数用 C 或者 C++ 编写，可以直接执行。 另一种是hook opcode，需要使用 zend 提供的 API zend_set_user_opcode_handler 来修改 opcode 的 handler 来实现。

opcode 目录下的每一个文件分别对应 hook 一种 opcode，xmark 扩展实现的 opcode 可以查看文档。以 echo 函数为例，在检测完 xss 后调用 xmark 扩展的 API xregister_opcode_callback 注册 opcode callback，该 API 底层实现是通过 hook XMARK_ECHO 来修改 opcode 的 handler，具体细节这里不做讨论。

sink

sink 目录下有五个子目录，分别对应五种不同的漏洞。通过在 sink 点观测来判断是否存在相应的漏洞并上报给 sentry 平台。

以 file_put_contents 函数为例，通过调用 /src/Utils.php 文件中的 prvd_check_path 函数检测 file_put_contents 函数的 fileaname 参数是否含有标记，从而判断是否存在任意文件写入。

一句话总结：通过检测常见危险函数的相关参数是否含有标记来判断是否存在相应的漏洞。

fuzzer

fuzzer.php：文件中有两个类，CurlHandler 和 Fuzzer

CurlHandler 类主要用来处理 curl 请求，有一个构造方法和一个析构方法。select 方法解析一个 curl 批处理句柄并释放。join 方法在限定请求时间内处理 curl 句柄。enqueue 方法拼接 url、method、headers、data字段为一个 HTTP 请求并加入 curl 请求队列中。

Fuzzer 类用于 payload 模式检测漏洞，构造方法设置 curl 选项并 new 一个 CurlHandler 类处理请求。add_value 方法给值加上所有 payload 得到一个数组，fuzz_value 对递归调用 add_value 方法给每个值加上 payload，start 方法启动 fuzz，fuzz_query、fuzz_body、fuzz_cookies、fuzz_files、fuzz_headers 方法调用 fuzz_value 方法对相应部分 fuzz，然后调用 make_request 方法生成请求。make_request 方法对不同的 HTTP 字段有相应的处理逻辑，data 字段根据编码方式有三种处理，分别是 application/x-www-form-urlencoded、application/json、multipart/form-data。

总结：fuzzer 对通过 file_get_contents('php://input') 获取的数据末尾加上 payload，然后在 sink 点观测是否触发漏洞，这就是项目的 payload 模式。需要注意的是，php://input 是个可以访问请求的原始数据的只读流（多为 POST 数据），当编码方式为 enctype=”multipart/form-data” 的时候 php://input 是无效的。

该项目的 fuzzer 只有 '"><xtanzi>./../xtanzi 这一个payload，实际应用中需要大量的 payload 来触发漏洞，因此可以扩充或重写 fuzzer。

taint & payload 模式对比

通过上图对比我们可以发现 taint 和 payload 模式处理流程相似，都是在 source 点打上标记然后在 sink 点观测，但是处理和性能上存在差异。

taint 和 payload 模式对比如下：

taint 模式 payload 模式
source 点标记 通过 xmark API 实现，对 HTTP 请求参数、HTTP header、敏感函数的返回值打上标记 通过 fuzzer 实现，仅对 php://input 的原始数据流打上 payload
filter 过滤处理 关注（taint模式难点就在于标记的处理） 不关注
sink 点观测 简单关注相应参数是否含有标记 需要通过各种条件判断是否触发漏洞
漏报率 低 高
误报率 高 低
是否污染数据 否 是
